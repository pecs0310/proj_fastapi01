# 2일차: Git & GitHub Flow 브랜치 전략 공부

## 1. Git Flow 전략
안정적인 릴리즈를 위해 엄격한 브랜치 구조를 가지는 전략입니다. 대규모 프로젝트나 정기적인 배포 주기가 있는 제품에 적합합니다.

* **main**: 제품으로 출시될 수 있는 가장 안정적인 상태의 코드를 관리하는 브랜치.
* **develop**: 다음 출시 버전을 개발하는 중심 브랜치. 기능 개발이 완료되면 이곳으로 병합됩니다.
* **feature**: 새로운 기능을 개발하는 브랜치. `develop`에서 분기하여 개발 후 다시 `develop`으로 병합됩니다.
* **release**: 배포를 준비하는 브랜치. 버그 수정 및 QA를 진행한 후 `main`과 `develop`에 병합됩니다.
* **hotfix**: 배포된 운영 버전(`main`)에서 긴급하게 발생한 버그를 수정하기 위한 브랜치.

## 2. GitHub Flow 전략
`main` 브랜치 하나를 중심으로 운영되는 단순하고 빠른 전략입니다. 상시 배포가 가능하고 지속적 통합/배포(CI/CD)가 잘 구축된 웹 서비스에 적합합니다.

* **main**: 언제든 배포 가능한 상태를 유지하는 유일한 중심 브랜치.
* **feature (or topic)**: 새로운 기능 개발이나 버그 수정을 위해 `main`에서 분기하는 브랜치. 브랜치명이 명확해야 합니다.
* **Pull Request (PR)**: 작업이 완료되면 `main` 브랜치로 병합하기 위해 피드백을 요청하고 코드 리뷰를 진행하는 과정.

## 3. 우리 팀의 선택 및 이유
* **선택한 전략**: GitHub Flow (혹은 팀원과 상의한 전략 기재)
* **이유**: API 개발 및 피드백을 빠르게 주고받으며 수시로 기능을 테스트하기 위해, 구조가 단순하고 병합 주기가 빠른 GitHub Flow가 이번 과제 진행에 더 효율적이라고 판단했습니다.
