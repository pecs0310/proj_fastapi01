### 🚀 [팀 공유용] FastAPI 프로젝트 Git 협업 가이드 & 충돌 방지 약속

우리 프로젝트는 **FastAPI 비동기 아키텍처**와 **MySQL**을 기반으로 진행됩니다. 팀원 각자가 `User`, `Patient`, `Record` 파트를 독립적으로 개발하면서 깃허브(GitHub)에서 코드 충돌 없이 부드럽게 협업하기 위해, 내일 스탠드업 미팅에서 공유할 핵심 충돌 포인트와 예방 수칙을 정리했습니다.

---

### 1. 4대 핵심 충돌 포인트 (SPOF) & 회피 전략

팀원 간 코드가 뒤엉키거나 개발 환경이 마비되는 단일 장애점(SPOF)을 방지하기 위한 보안관 규칙입니다.

#### 🚨 충돌 1: 로컬 MySQL 비밀번호 동기화 오류

* **원인:** 팀원마다 본인 컴퓨터에 설치된 MySQL의 root 패스워드가 다릅니다. 공용 설정 파일인 `app/core/config.py`를 직접 수정해서 깃허브에 올리면, 다른 팀원의 로컬 서버가 일제히 DB 연결 에러를 뱉으며 멈춥니다.

  > 😉😉😉 아래는 직접 화면공유해서 진행 😉😉😉
* **회피 전략:** `config.py`는 절대 수정하여 푸시하지 않습니다. 프로젝트 루트에 각자 **`.env`** 파일을 만들고 본인의 비밀번호를 적어 관리합니다. (이 파일은 `.gitignore`에 의해 깃허브에 올라가지 않으므로 안전합니다.) 

#### 🚨 충돌 2: 단일 파일 동시 수정으로 인한 머지 충돌 (Merge Conflict) >> 협의완료

* **원인:** 모든 DB 설계도를 하나의 `models.py`에 짜거나, 엔드포인트를 하나의 `main.py`에 몰아서 작성하면 깃허브에 코드를 합칠 때 100% 충돌이 발생합니다.
* **회피 전략:** 철저한 **도메인 기반 파일 격리**를 적용합니다. 각자의 담당 폴더와 파일 안에서만 코딩합니다.
* **User 담당:** `models/user.py`, `apis/user.py`, `schemas/user.py`
* **Patient 담당:** `models/patient.py`, `apis/patient.py`, `schemas/patient.py`
* **Record 담당:** `models/record.py`, `apis/record.py`, `schemas/record.py`



#### 🚨 충돌 3: 파이썬 캐시 및 로컬 데이터의 레포지토리 오염

* **원인:** 파이썬 실행 시 자동 생성되는 `__pycache__` 폴더나 로컬 테스트용 미디어 파일들이 깃허브에 올라가면 동기화 속도가 저하되고 파일 경로 꼬임 현상이 발생합니다.
* **회피 전략:** 프로젝트 Owner가 세팅한 **`.gitignore`** 설정을 절대 변경하지 마세요. 만약 에디터 설정 파일(`.vscode` 등)이 자꾸 커밋에 잡힌다면 스탠드업시간에 이야기 해봅시다. 조교님소환이 필요한 사안인지..

#### 🚨 충돌 4: `main` 브랜치 직접 푸시로 인한 전체 시스템 크래시 >> 협의완료

* **원인:** 검증되지 않은 문법 에러가 포함된 코드가 `main` 브랜치에 직접 올라가면, 전체 팀원의 로컬 환경이 동시에 구동 불능 상태에 빠집니다.
* **회피 전략:** `main` 브랜치 직접 푸시는 금지됩니다. 각자 기능 브랜치를 파서 작업한 뒤, 코드 리뷰(PR)를 거쳐 합칩니다. 
  > 어차피 하고 싶어도 안됩니다. 프로텍션 걸려 있어서...ㅋㅋ

### 2. 우리가 지킬 데일리 Git 워크플로우

출근 직후와 퇴근 전, 아래 4단계 루틴만 준수하면 깃으로 인해 스트레스받을 일이 없습니다.

> **💡 매일 아침 출근 직후 (최신 코드 싱크)**
> 1. `main` 브랜치로 이동: `git checkout main`
> 2. 팀원들이 올린 최신 코드 원격에서 가져오기: `git pull origin main`
> 
> 

```text
[새 작업 시작 시 브랜치 생성 규칙]
git checkout -b feature/user    <- User 담당자
git checkout -b feature/record  <- Record 담당자

```

> **💡 기능 개발 완료 후 퇴근 전 (코드 제출)**
> 1. 내 도메인 파일들만 스테이징: `git add app/models/user.py app/apis/user.py ...`
> 2. 직관적인 커밋 메시지 작성: `git commit -m "feat: 유저 회원가입 API 구현"`
> 3. 내 브랜치에 푸시: `git push origin feature/user`
> 4. 깃허브 레포지토리 페이지에 접속하여 **Pull Request (PR)** 생성 후 리뷰어 지정 
> 
>
