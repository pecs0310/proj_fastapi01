# 어떻게 손 안대고 코드를 관리할 것인가 - Git branch 전략.

원래부터 개념은 알고 있었지만, 혼자서만 개발을 하다보니 크게 신경쓸 일이 없었던 깃브랜치 전략을 정리했다. 깃브랜치 전략은 이를테면, 여러사람이 하나의 프로젝트에 관여하고 있을 때 코드베이스를 어떻게 '손 안대고 관리할 것인가'에 대한 '입장표명'이라고 할 수 있다. 완벽하지만 복잡하게, 빠르지만 아슬아슬한 양 극단에 있는 2가지 '입장'을 소개하고 각 어울리는 상황에 대해서 정리해보도록 하겠다.


## Git v. GitHub Flow

위에서 말한 '양 극단의 입장'이란 **Git-Flow**와 **GitHub-Flow**인데, 이 둘은 출생배경과 그에따른 철학 자체가 완전히 다르다. 따라서 **시대적 배경, 배포(Release) 환경, 그리고 팀의 커뮤니케이션 방식**에서 다음과 같은 차이점이 발생하게 된다.

---
### 1. 출신 배경의 차이: 무엇이 이들을 만들었는가?

두 전략의 결정적 차이는 **"우리가 만드는 소프트웨어는 얼마나 자주, 어떤 방식으로 사용자에게 전달되는가?"**라는 질문에서 출발한다.

| 구분 | Git-Flow | GitHub-Flow |
| --- | --- | --- |
| **제안자/시기** | 빈센트 드라이센 (2010년) | 스콧 차콘 / GitHub (2011년) |
| **주요 배경** | **정기 배포 기반의 패키지/앱 환경** | **수시 배포 기반의 웹 서비스 환경** |
| **핵심 사상** | "완벽하게 검증된 버전을 정기적으로 출시하자." | "살아있는 서비스는 하루에도 수십 번씩 배포되어야 한다." |

### 1) Git-Flow의 배경: 패키지 소프트웨어 시대

2010년 당시에는 스마트폰 앱이나 설치형 소프트웨어(CD, 패키지)처럼 **한 번 배포하면 되돌리기 어렵거나, 배포 주기가 몇 주~몇 달 단위로 긴 제품**이 주를 이루었다. 즉, 오픈소스나 대규모 엔터프라이즈의 **중앙집중식 통제 모델**에 가깝다. 

당연히 버전 관리(v1.0, v1.1 등)가 매우 중요했고, 출시 전에 기획, QA, 마케팅 일정을 맞춰 "이번 시즌 출시작"을 꼼꼼히 검증해야 했다. 따라서 여러 브랜치를 촘촘하게 엮어 실수를 방지하는 엄격한 구조가 필요했던 것. 

메인 관리자(Maintainer)가 수많은 기여자의 코드를 엄격하게 검증하여 특정 타이틀(v1.0, v2.0)로 릴리즈 주기를 제어하려는 목적을 가진다. 릴리즈 노트 작성이 중요하고 상용 배포의 '타이밍'은 경영진이나 마케팅 부서 등 '양복쟁이'들의 손에 달려 있었다. 1980년에 깃플로우가 일반화되어 있었다면 그 유명한 '아타리쇼크'도 예방할 수 있었으리라(아닌가?).  


### 2) GitHub-Flow의 배경: SaaS 및 웹 서비스 시대, 분산형 '자율과 책임'

반면 GitHub은 전 세계 개발자가 매 초 코드를 수정하고 공유하는 SaaS(서비스형 소프트웨어) 회사다. 웹 서비스는 패키지와 달리 **코드를 서버에 반영하는 즉시 사용자가 최신 버전을 쓰게 된다.** 버그가 있으면 10분 만에 고쳐서 다시 배포하면 된다. 

이 환경에서는 복잡한 릴리즈 절차보다 "기능 하나 만들었으면 빠르게 테스트하고 지금 바로 서비스에 반영(배포)하자"는 속도가 생명이었으니까. 이들은 중앙 관리자의 승인을 기다리는 병목 자체를 죄악으로 여긴다. 

기능(Feature) 단위로 쪼개진 변경 사항이 다른 기능에 영향을 주지 않도록 마이크로서비스 아키텍처(MSA)나 느슨한 결합(Loose Coupling)을 전제로 움직이는 시스템만이 가질 수 있다. "망가지면 바로 고쳐서 다시 밀어 넣는다(Fail Fast, Fix Fast)"는 철학이 기저에 깔려 있다. Git-Flow가 웹 환경에는 너무 무겁고 느려서 만들어진 대안이므로 어찌보면 당연한결과랄까.

---


### 2. 구조적 차이점 발생 원인: 5개 브랜치 vs 1개 브랜치

출신 배경이 다르다 보니, 관리하는 브랜치의 체계와 복잡도에서 극단적인 차이가 발생한다.

### 1) Git-Flow: 엄격한 역할 분담과 5단계 구조

Git-Flow는 언제나 유지되는 핵심 브랜치 2개와 살다 없어지는 보조 브랜치 3개를 쓴다.

* **핵심 브랜치:** `main`(실제 서비스 중인 안정적인 코드), `develop`(다음 버전을 위해 개발 중인 코드)
* **보조 브랜치:** `feature`(기능 개발), `release`(출시 전 최종 QA/버그 수정), `hotfix`(서비스 중 긴급 버그 수정)

> **구조적 결과:** 기능 하나를 만들면 `feature` ➔ `develop` ➔ `release` ➔ `main` 순서로 계단을 밟듯이 코드가 이동하게 된다. 검증은 완벽할지 몰라도, 관리 비용이 크고 머지(Merge)가 복잡하다.

### 2) GitHub-Flow: 극단적 단순화와 상시 배포

GitHub-Flow는 복잡한 단계를 다 걷어내고 딱 하나만 기억한다. 
> **"`main` 브랜치는 언제나 배포 가능한 상태여야 한다."**

* **핵심 브랜치:** `main` 끝.
* **보조 브랜치:** `feature/*` 또는 자유로운 이름의 임시 브랜치.

> **구조적 결과:** `main`에서 브랜치를 파서 기능을 만든 뒤, 테스트가 끝나면 **즉시 `main`으로 합치고 그 자리에서 바로 서버에 배포**. `develop`이나 `release` 브랜치 단계를 거치지 않으므로 피드백 순환이 압도적으로 빨라진다.



## 아키텍처의 깊이 = 검증 파이프라인의 차이

Git-Flow는 정교하고, GitHub-Flow는 극도로 단순하다. 이를 엔지니어링 관점에서 쪼개보면 인프라와 파이프라인의 요구사항이 완전히 달라지는데...

### 1) Git-Flow: '환경(Environment)' 기반의 다단계 격리

Git-Flow의 구조는 인프라 환경(`Local ➔ Dev ➔ Staging ➔ Prod`)과 1:1로 매핑된다.

* **신뢰 구축 방식:** 코드가 상용에 가기 전 `develop`과 `release`라는 두 번의 완충지대를 둔다는 점은 상술했다. 이 구간에서 사람이 개입하는 QA, 인수 테스트(UAT), 스테이징 검증이 물리적 시간 차를 두고 일어나게 된다.
* **장점:** CI/CD 자동화 수준이 낮아도, **인간 검증 프로세스**가 촘촘하여 대형 배포 사고를 막을 수 있다는 것이 가장 큰 장점이다.
* **단점:** 개발 주기가 수개월인 경우, `feature` 브랜치들이 `develop`에 장기간 쌓이면서 거대한 '머지 지옥(Merge Hell)'과 형상 동기화 실패를 유발한다.

### 🌐 GitHub-Flow: '테스트 자동화(CI/CD)' 기반의 지속적 신뢰

GitHub-Flow는 중간 완충지대(`develop`, `release`)를 완전히 제거했다.

* **신뢰 구축 방식:** 완충지대가 없기 때문에 "코드 리뷰(PR)"와 "100% 신뢰할 수 있는 자동화 테스트(CI/CD)"가 그 역할을 대신한다. PR을 올린 순간 테스트 러너가 빌드, 유닛 테스트, 통합 테스트를 자동으로 수행하여 안전성을 증명해야 한다(즉, [ 자동 ] 테스트시스템의 존재가 중요한 전제이다).
* **장점:** 병목이 없습니다. 리뷰와 테스트가 통과되면 즉시 상용 배포가 가능하므로 비즈니스 피드백 속도가 극대화된다.
* **단점:** 자동화 테스트 커버리지가 낮거나 파이프라인이 부실하면, 결함이 있는 코드가 어떠한 제동 장치도 없이 `main`을 통해 실시간으로 사용자에게 노출된다.

## 결론
위에서 살펴본 두 방식의 차이는 "변경 사항(Change)을 검증하고 상용 환경(Production)에 전달하는 신뢰도 모델(Trust Model)을 어떻게 설계했는가"라는, 위험을 관리하는 시선의 차이에 의한다. 

* **Git-Flow의 SPOF 관리:** 인간의 실수를 시스템(단계)으로 막고 여러 겹의 성벽(`develop`, `release`)을 쌓아 `main`을 보호하므로 안전하지만, 성벽 사이에서 코드 싱크가 맞지 않아 발생하는 "머지 지옥(Merge Hell)"이 또 다른 장애점이 될 수 있다(하지만 이건 일센스 넘치는 개발팀장만 있다면 최선의 전략이라고도 생각한다).

* **GitHub-Flow의 SPOF 관리:** 단계를 없앤 대신, 자동화(CI/CD)와 Pull Request(코드 리뷰)에 목숨을 건다(걸어야 한다). 코드가 `main`에 합쳐지기 전에 자동 테스트 스크립트가 버그를 완벽히 걸러내야 하며, 팀원 간의 꼼꼼한 코드 리뷰가 필수적이다(하지만 믿을 수는 없지). 이 자동화 테스트 망이 부실하면 `main`이 쉽게 무너진다. 결국 위의 방법을 사용하던 회사의 '일센스 있는 개발팀장'이 야근이 지겨워지면 도달하게 되는 경지가 된다.


## 우리팀에 맞는 관리방식 선택 매트릭스

두 방식의 극단적인 장단점 때문에, 최근 실무에서는 이 둘을 절충한 **GitLab-Flow**나 **Trunk-based Development** 같은 변형 모델을 쓰기도 합니다. 현재 팀의 상황을 진단하기 위한 체크리스트는 다음과 같다.

| 의사결정 기준 | Git-Flow 채택 | GitHub-Flow 채택 |
| --- | --- | --- |
| **배포 주기** | 정기 배포 (주 1회, 격주, 혹은 월간) | 상시 배포 (하루에도 여러 번 가능) |
| **인프라 환경** | 개발계, 검증계, 운영계 등 서버 분리가 엄격함 | 컨테이너 기반으로 동적 테스트 환경 구성 가능 |
| **자동화 수준** | 빌드/테스트 자동화가 없거나 불완전함 | CI/CD 파이프라인과 테스트 커버리지가 높음 |
| **팀 규모 및 역량** | 인원이 많고 역할(QA, 기획, 개발)이 분리됨 | 소규모 정예이거나 개발자 주도의 DevOps가 정착됨 |

두세 명의 협력자와 함께 빌드업하는 현재 단계에서는 **GitHub-Flow의 단순함**을 가져가되, 파이프라인이 빈약할 수 있으므로 '동료 1인 이상의 승인(Approve) 후 머지'라는 인간적인 통제 장치(Harness)를 엮는 것이 가장 현실적이고 효율적인 최적해(Optimal Path)인 듯하다. 대규모 아키텍처가 아닌 이상, 통제 비용이 개발 비용을 초과하게 만들지 않는 것이 중요하기 때문이다(그리고 1명정도의 동의는 있는게 형태상으로도 뽀대가 나는 거 같다).


### 우리는 CI/CD 파이프라인이 없는데요?
자동화 파이프라인(CI/CD)을 구축할 인력이나 기술적 여유가 없는 것은 초기 팀이나 리포지토리에서 **매우 당연하고 자연스러운 현상**입니다. 인프라 공사에 리소스를 뺏기기보다 비즈니스 로직과 제품 자체에 집중하는 것이 훨씬 현명한 전략이다.

대신 시스템 자동화 대신 **'인간의 프로세스(Human Process)'로 파이프라인을 대체하는 최적화 경로**를 선택하면 된다. 시스템적 재갈(Harness) 대신 규칙을 통한 통제를 가하는 것.


#### 대안 1 : CI/CD가 없을 때의 '수동 파이프라인' 아키텍처

자동화 툴이 하던 일을 개발 프로세스 안에서 **규칙과 체크리스트**로 녹여내면 GitHub-Flow의 단순함을 그대로 유지하면서도 안정성을 확보할 수 있다.

```
[작업자] 브랜치 생성 및 기능 개발
   ⬇
[수동 검증] 본인 로컬(Local) PC에서 빌드 및 정상 작동 최종 확인 (필수)
   ⬇
[GitHub] main 브랜치로 Pull Request(PR) 생성
   ⬇
[동료 리뷰] 다른 협력자가 코드를 눈으로 검증(Code Review) 후 Approve
   ⬇
[소유자 병합] 리포지토리 소유자(본인)가 최종 확인 후 main에 Merge
   ⬇
[수동 배포] main 코드를 서버에 수동으로 반영 (git pull 등)

```

> **수동 파이프라인의 단점과 극복방법** : 
이 모델의 유일한 단점은 **"협력자의 로컬 환경과 실제 서버 환경의 불일치"이다**. "제 컴퓨터에서는 잘 되는데요?"라는 -서로 빡치는- 상황이 반복되면 서버도 무너지고 신뢰도 함께 붕괴되기 때문. 따라서 소스 코드에 데이터베이스 연결 정보(DB URL), API Key 같은 환경 변수(`env`)를 절대 하드코딩하지 못하게 통제하고, 환경 변수는 서버와 로컬이 각자 관리하도록 규칙을 박아두는 것이 중요하다.


#### 대안 2 :  비용 제로, 학습 제로: 가성비 대체재 - Github 무료기능 이용
파이프라인을 직접 구축하지 않더라도, GitHub가 자체적으로 제공하는 무료 기능을 약간만 활용하면 완벽하진 않아도 80%의 안정성을 확보할 수 있다.

### 1) GitHub Issues의 템플릿화 (수동 체크리스트 강제)

협력자가 코드를 합쳐달라고 PR(Pull Request)을 보낼 때 아무 말 없이 코드만 툭 던지지 못하도록 작성 양식(Template)을 강제하고, PR 본문에 아래와 같은 수동 체크리스트를 적도록 팀 규칙을 추가한다.

```markdown
# 배포 전 수동 체크리스트
- [ ] 로컬 환경에서 에러 없이 정상적으로 빌드/구동되는지 확인했습니까?
- [ ] 이번 수정으로 인해 영향을 받는 다른 기능이 있습니까? (있다면 명시)
- [ ] 데이터베이스 구조(Schema) 변경이 필요합니까?
```

이 박스에 모두 체크를 해야만 리뷰어가 코드를 봐주는 [ 문화적 ] 제약(Harness)을 선언하는 것이다.

### 2) 딱 한 줄의 자동화: GitHub Actions 맛보기 (선택 사항)
복잡한 배포(CD)는 건너뛰더라도, "코드가 깨지는지(Build Error)만 잡아내는 수준"의 자동화는 클릭 몇 번으로 가능하다고 한다.
GitHub 리포지토리 상단의 **Actions** 탭에 들어가면 내 프로젝트 언어(Python, Node.js 등)에 맞는 기본 템플릿을 추천해 준다. `Configure` 버튼을 누르고 저장만 하면, 협력자가 코드를 올릴 때마다 "컴파일은 정상적으로 되는가?"를 GitHub가 대신 검사해 주므로 배포까지 안 가더라도 이 정도만 있으면 "빌드도 안 되는 쓰레기 코드"가 메인 브랜치에 섞이는 것은 완벽히 차단된다.

### 3) 결론

사실 인프라 능력이 없을 때는 원론으로 돌아가 "더 꼼꼼하게 소통하고, 더 작게 쪼개서 자주 합치는 것"이 유일한 해법이다. 동료들은 일주일 동안 짠 거대한 코드 덩어리가 아니라 "하루 단위, 혹은 기능 하나 단위로 아주 잘게 쪼개서 PR을 올리라"고 약속하고, (형식적) 파이프라인이 없어도 눈으로 검증(Code Review)할 수 있는 인지적 범위 안으로 통제선이 확대되지 않도록 제어해야 한다.

지금 동료 2명과 함께 시작하는 프로젝트는 **수시로 업데이트되는 웹 서비스나 토이 프로젝트에 가까우므로 GitHub-Flow가 훨씬 유리**하다. 인원이 적을 때 Git-Flow를 도입하면, 코드 짜는 시간보다 브랜치 옮겨 다니고 머지 꼬인 거 푸는 데 시간을 더 많이 쓰게 되어 비효율적이기 때문이다.

> 다만, GitHub-Flow를 쓰더라도 앞서 언급한 것처럼 **`main` 브랜치에 'Require a pull request before merging' 규칙은 반드시 걸어** 최소한의 안전장치를 확보해야 한다.
