# Git Branch 전략 정리

## 1. 브랜치 전략이란?

여러 명의 개발자가 동시에 작업할 때, 서로의 코드에 영향을 주지 않고  
독립적으로 기능을 개발하고 안전하게 합치기 위한 **브랜치 관리 규칙**이다.

대표적인 전략으로 **Git Flow**와 **GitHub Flow**가 있다.

---

## 2. Git Flow

### 개념
복잡한 릴리즈 주기를 가진 프로젝트에 적합한 전략으로,  
**5가지 브랜치**를 역할에 따라 명확히 분리하여 운영한다.

### 브랜치 구조

| 브랜치 | 역할 |
|--------|------|
| `main` (or `master`) | 실제 배포되는 안정적인 코드 |
| `develop` | 다음 릴리즈를 위한 개발 통합 브랜치 |
| `feature/기능명` | 새로운 기능 개발 (develop에서 분기) |
| `release/버전` | 배포 전 최종 테스트 및 버그 수정 |
| `hotfix/이슈명` | 배포된 main의 긴급 버그 수정 |

### 흐름 요약

```
main
 └── develop
       ├── feature/login       ← 기능 개발
       ├── feature/signup      ← 기능 개발
       └── release/1.0.0       ← 배포 준비
             └── hotfix/bug    ← 긴급 수정 (main & develop 반영)
```

### 장점
- 브랜치 역할이 명확하여 대규모 팀에서 관리가 용이
- 릴리즈 버전 관리가 체계적

### 단점
- 브랜치 수가 많아 복잡하고, 소규모 팀에는 오버엔지니어링이 될 수 있음
- 배포 주기가 짧은 프로젝트에는 부적합

---

## 3. GitHub Flow

### 개념
Git Flow보다 단순화된 전략으로,  
**`main` 브랜치 하나**와 **`feature` 브랜치만**으로 운영한다.  
빠른 배포와 지속적인 통합(CI/CD)에 적합하다.

### 브랜치 구조

| 브랜치 | 역할 |
|--------|------|
| `main` | 항상 배포 가능한 상태를 유지 |
| `feature/기능명` | 기능 개발 후 main에 PR로 병합 |

### 흐름 요약

```
1. main에서 develop 브랜치 분기
2. 기능 개발 후 커밋 & 푸시
3. Pull Request(PR) 생성
4. 팀원 코드 리뷰
5. main에 Merge
6. 바로 배포
```

```
main ──────────────────────────────────────► (항상 배포 가능)
       └── feature/user-api ──► PR ──► Merge
```

### 장점
- 구조가 단순하여 소규모 팀이나 빠른 배포 환경에 적합
- PR 기반 코드 리뷰가 자연스럽게 이루어짐

### 단점
- 브랜치 역할 구분이 약해 대규모 프로젝트 관리에는 한계
- 릴리즈 버전 관리가 명확하지 않음

---

## 4. Git Flow vs GitHub Flow 비교

| 항목 | Git Flow | GitHub Flow |
|------|----------|-------------|
| 브랜치 수 | 5가지 (main, develop, feature, release, hotfix) | 2가지 (main, develop) |
| 복잡도 | 높음 | 낮음 |
| 배포 주기 | 정기 릴리즈 | 수시 배포 |
| 적합한 팀 규모 | 중·대규모 | 소규모 |
| CI/CD 친화성 | 보통 | 높음 |

---

## 5. 우리 팀이 선택한 전략

> **GitHub Flow**

### 선택 이유
- 팀원이 3명으로 소규모 프로젝트에 적합
- feature 브랜치 → PR → main Merge 흐름이 단순하고 명확
- 빠른 피드백과 코드 리뷰가 가능

### 우리 팀 브랜치 네이밍 규칙

```
feature/기능명         ex) feature/user-api
hotfix/버그내용        ex) hotfix/login-error
docs/문서명            ex) docs/team-rules
```

### 커밋 메시지 규칙

```
feat: 새로운 기능 추가
fix: 버그 수정
docs: 문서 작성 및 수정
refactor: 코드 리팩토링
chore: 기타 설정 변경
```
